开发者培养

## PS:重点看标红的部分 培养目标

StarLandAI 的开发者培养，不是为了把成员训练成“会调几个模型 API 的使用者”，而是要培养能够围绕真实问题完成需求判断、技术选型、开发实现、部署上线、演示汇报与复盘迭代的 AI Agent 工程师。

最终要形成两类能力：

1. 会用 AI Agent：能够熟练调用现成模型、工作流、搜索、RAG、自动化工具，提高学习、调研、写作、分析和协作效率。
2. 会做 AI Agent：能够把一个实际问题拆成可执行的工程任务，做出可运行、可部署、可演示、可复用的产品原型或 MVP。

## 核心原则

### 1. 差异化

不要默认去做大厂和顶级科研机构最卷的通用赛道，而是优先寻找更适合学生团队切入的垂直场景、具体流程和结构化需求。

例如，与其泛泛地做“万能 agent”，不如做：

- 面向劳动者维权的 `ClaimClaw`
- 面向学习成长与组织协作的 `Starsea`
- 面向真实信息监测、内容生产、图形编辑的垂直 agent

差异化的本质，不是故意做冷门，而是选择一个“需求明确、数据可得、可做出体验差异、能形成长期积累”的方向。

### 2. 工程化

项目不能停留在 `python main.py`、一段 prompt、或一个临时 notebook。

工程化至少意味着：

- 需要有明确的产品形态：网页、Bot、插件、脚本工具、服务端 API、知识库工作流等
- 需要有清晰的仓库结构、README、依赖说明和启动方式
- 需要能部署：本地可运行，最好能进一步部署到服务器、容器或在线服务
- 需要能演示：至少要有一条完整的使用链路和 demo 视频
- 需要能持续迭代：有 issue、版本记录、路线图和复盘

### 3. 场景化

所有项目都必须回答以下问题：

- 这个 agent 到底解决谁的问题？
- 它在什么场景下被使用？
- 为什么用户愿意持续用它，而不是只看一次 demo？
- 它和普通聊天机器人、搜索引擎、人工流程相比，优势在哪里？

### 4. 可验证

开发者不能只展示“我觉得可以”，而要尽量展示：

- 输入与输出样例
- 效果对比
- 数据来源与引用
- 指标或案例
- 用户反馈或自测结论

### 5. 可复用

一个好项目不应该只能由作者本人运行。至少要做到：

- 别人能按文档跑起来
- 别人能理解系统结构
- 别人能接着扩展模块
- 别人能复用你的组件、Prompt、数据结构或前后端骨架

## 工程师计划（Engineer Plan）

工程师计划分为两部分，一针对学习，一针对项目。

### 学习

开发者必须每周汇报学习进度。汇报不应只写“看了什么”，还应写清楚：

- 本周学习主题与进度
- 关键收获
- 已完成的实验或尝试
- 遇到的问题
- 下周计划

所有学习汇报都应托管上云，并把链接交给信息保障部整理归档。

### 个人项目

从本周起（4.21开始），每位开发部的成员都必须创建自己的个人项目并进行维护（除实习人员），对兼任规划部的同学不做强制要求（自愿展示，不参与credit榜）。

2.所有项目必须要有

git版本管理（命令行，不允许长期使用图形化工具）,

github仓库，

文档，

#### 汇报要求

开发者每两周汇报一次个人项目。（同样要求将以下内容托管上云）

首次汇报提交以下材料：(建议ppt)

1. GitHub 仓库地址
2. 项目概述
3. MVP 目标
4. 当前技术路线
5. 第一版时间计划

后续汇报至少应包含：

- 已完成模块
- 当前阻塞点
- 新增截图或 demo
- 下一阶段计划

### 文档归档规范

每个项目都应该逐步沉淀以下文档：

- 项目概述
- 架构图 / 流程图
- API 或模块说明
- 部署文档
- 测试记录
- demo 视频
- 复盘总结

托管上云的规范：使用飞书的云盘功能，托管在云盘上，并在知识库或我的文档库中为项目创建对应空间

## Credit 积分榜制度

开发者每人每周有 `1 credit`，规划师每周有 `5 credits`。组内建立 credit 榜单。

在项目、学习进度汇报上，对其他人的项目进行投票。

【会在汇报前提供投票应用，credit可用于他人的学习分享投票（如你觉得他人分享的笔记很有价值）

   或对别人的项目投票】

credit 的意义不只是排名，而是将成员的投入与资源分配挂钩。积分较高的成员，在以下资源上应享有更高优先级：

- 先进 CLI 编程工具或等额报销
- 云服务器
- GPU 调用额度
- 组织项目协作机会
- 项目收益分配优先级

后续细则可根据实际推进情况再逐步明确，但原则应尽量做到公开、可追踪、可解释。

## 开发者基本要求

开发者至少应持续补齐以下基本开发能力：

- 使用 Git / GitHub 进行版本管理
- 阅读官方文档、开源仓库与 API 文档
- 能独立搭建 Python / Node.js 基础开发环境
- 能使用 Docker、部署脚本或云服务器完成基础部署
- 能把项目过程记录成文档，而不是只留在聊天记录里
- 能录制完整 demo，并进行讲解汇报

## 选题标准

一个合格的个人项目，建议同时满足以下条件：

1. 有真实需求，而不是为了“做个 agent”而做
2. 有明确用户或使用对象，哪怕一开始只服务组织内部
3. 有可接入的数据源、文档源或流程入口
4. 在 8 到 12 周内有机会做出第一版 MVP
5. 能做出演示链路，而不是只能口头介绍
6. 后续能沉淀为组织资产，例如组件、知识库、模板、数据结构或部署经验

## 推荐培养路径

### 第一阶段：会用

先建立 AI 工具使用能力：

- 会做资料检索、信息整理、结构化总结
- 会写清晰 prompt
- 会用模型完成代码解释、文档整理、任务拆分、测试辅助
- 会把 AI 用在学习、研究、项目推进而不是只停留在聊天

### 第二阶段：会做

开始做最小闭环：

- 明确用户问题
- 画出功能链路
- 接入模型、搜索、抓取或知识库
- 做出最小可运行版本
- 用真实样例反复测试

### 第三阶段：会部署

把项目从“能跑”推进到“能交付”：

- 补 README、配置说明、环境变量说明
- 用 Docker 或脚本封装启动
- 接入云服务器、域名、在线访问或 Bot
- 做监控、日志和异常处理

### 第四阶段：会协作

真正进入工程团队状态：

- 用 issue / milestone 拆任务
- 用 PR 管理改动
- 用文档沉淀接口、结构和路线图
- 用周报、双周汇报和 demo 对齐进度

## 交付标准

每个个人项目至少应有以下交付物：

1. GitHub 仓库地址
2. README，写清楚项目目标、技术栈、运行方式、当前进度
3. 一页式项目概述，写清楚用户、场景、核心价值、主要模块
4. 至少一份架构说明或流程图
5. 至少一条完整 demo 视频
6. 可运行的本地脚本、Web 页面、Bot、API 或部署地址
7. 迭代记录：周报、版本更新、问题清单或复盘

## 最终目标

StarLandAI 希望培养的开发者，不是只会调用模型的人，而是能把 AI 真正做进产品、工作流和组织能力中的人。

一个成熟开发者的标志，不是说自己“懂 AI”，而是能持续交付：

- 可运行的项目
- 可复用的资产
- 可展示的成果
- 可延续的工程能力
